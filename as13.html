1 검증 계획과 통과 기준

무엇을 어떻게 확인할지 정한다

개념 상세 설명: 검증 계획은 어떤 기능을 어떤 입력으로 시험하고 어떤 결과가 나오면 통과인지 미리 정하는 문서이다. 테스트 목적, 입력, 예상 결과, 실제 결과와 판정을 포함하면 시험 과정이 일관되고 누락을 줄일 수 있다.

test_plan = {
    "기능": "점수 평균 계산",
    "목적": "숫자 목록의 평균이 정확한지 확인",
    "입력": [80, 90, 100],
    "예상 결과": 90.0,
    "통과 기준": "실제 결과가 예상 결과와 같음"
}

for key, value in test_plan.items():
    print(f"{key}: {value}")
코드 상세 설명: 테스트 전에 확인할 기능과 예상 결과를 명확히 적는다. 통과 기준이 정해져 있어야 성공과 실패를 객관적으로 판단할 수 있다.
[개념 1] 검증 계획에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ⑤ — 검증 계획은 목적, 입력, 예상 결과와 통과 기준을 미리 정의한다.

2 테스트 케이스의 구성

입력·절차·예상 결과·판정

개념 상세 설명: 테스트 케이스는 하나의 구체적인 시험 사례이다. 식별 번호, 사전 조건, 입력, 실행 절차, 예상 결과, 실제 결과와 통과 여부를 포함하면 오류를 다시 확인하기 쉽다.

def grade(score):
    return "통과" if score >= 60 else "보완"

case = {
    "번호": "TC-01",
    "입력": 75,
    "예상": "통과"
}

actual = grade(case["입력"])
case["실제"] = actual
case["판정"] = actual == case["예상"]

print(case)
코드 상세 설명: 테스트 입력 75에 대한 예상 결과와 실제 결과를 비교한다. 판정값이 True이면 테스트가 통과한 것이다.
[개념 2] 테스트 케이스에 포함할 내용으로 옳은 것 3개를 고르시오.
정답: ①, ②, ④ — 테스트 사례는 입력, 기대 결과, 실제 결과와 판정을 남겨야 한다.

3 정상 사례

일반적인 사용 조건 확인

개념 상세 설명: 정상 사례는 사용자가 올바른 형식과 일반적인 범위의 값을 입력하는 경우이다. 기본 기능이 요구사항대로 작동하는지 확인하는 출발점이지만 정상 사례만으로 충분하지는 않다.

def calculate_average(values):
    if not values:
        return None
    return sum(values) / len(values)

normal_cases = [
    ([80, 90, 100], 90.0),
    ([10, 20], 15.0),
    ([7], 7.0)
]

for values, expected in normal_cases:
    actual = calculate_average(values)
    print(values, actual, actual == expected)
코드 상세 설명: 일반적인 숫자 목록을 입력해 평균 계산이 정확한지 확인한다. 각 사례는 입력과 예상 결과의 쌍으로 구성된다.
[개념 3] 정상 사례에 대한 옳은 설명 3개를 고르시오.
정답: ②, ③, ⑤ — 정상 사례는 기본 동작을 확인하지만 다른 사례도 함께 필요하다.

4 경계 사례

범위의 끝과 바로 바깥을 시험

개념 상세 설명: 경계 사례는 허용 범위의 최솟값·최댓값과 그 바로 안팎의 값을 시험한다. 비교 연산자 오류나 인덱스 범위 오류는 경계에서 자주 나타난다.

def valid_score(score):
    return 0 <= score <= 100

boundary_cases = [
    (-1, False),
    (0, True),
    (1, True),
    (99, True),
    (100, True),
    (101, False)
]

for value, expected in boundary_cases:
    actual = valid_score(value)
    print(value, actual, actual == expected)
코드 상세 설명: 허용 범위 0~100의 경계값과 바로 바깥값을 함께 시험한다. <와 <=를 잘못 사용한 오류를 쉽게 발견할 수 있다.
[개념 4] 경계 사례에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ④ — 범위 끝과 인접 값을 시험해 경계 오류를 찾는다.

5 예외 사례

잘못된 입력에도 안전하게 대응

개념 상세 설명: 예외 사례는 빈 값, 잘못된 자료형, 결측값, 음수, 지나치게 큰 값처럼 정상 흐름에서 벗어난 입력이다. 프로그램이 중단되지 않고 명확한 오류 메시지나 안전한 기본값을 제공하는지 확인한다.

def parse_age(raw):
    try:
        age = int(raw)
    except (TypeError, ValueError):
        return None, "나이는 정수로 입력하세요."

    if not 0 <= age <= 120:
        return None, "나이는 0~120 범위로 입력하세요."

    return age, "정상 입력"

exception_cases = ["", "열여섯", None, -1, 999]

for value in exception_cases:
    print(value, "→", parse_age(value))
코드 상세 설명: 변환 실패와 범위 오류를 구분하여 처리한다. 예외 입력이 들어와도 프로그램 전체가 갑자기 종료되지 않도록 설계한다.
[개념 5] 예외 사례에 대한 옳은 설명 3개를 고르시오.
정답: ①, ②, ⑤ — 예외 입력에 대한 안전한 처리와 안내를 검증한다.

6 자동 테스트와 assert

예상 결과를 코드로 반복 확인

개념 상세 설명: 자동 테스트는 여러 사례를 코드로 실행해 실제 결과와 예상 결과를 빠르게 비교하는 방법이다. assert를 이용하면 조건이 다를 때 즉시 실패 위치를 알 수 있다.

def clamp(value, low, high):
    return max(low, min(value, high))

assert clamp(5, 0, 10) == 5
assert clamp(-1, 0, 10) == 0
assert clamp(15, 0, 10) == 10
assert clamp(0, 0, 10) == 0
assert clamp(10, 0, 10) == 10

print("모든 테스트 통과")
코드 상세 설명: 정상값, 범위 아래·위 값과 양쪽 경계값을 한 번에 시험한다. 함수 수정 후에도 같은 테스트를 반복 실행할 수 있다.
[개념 6] 자동 테스트에 대한 옳은 설명 3개를 고르시오.
정답: ②, ④, ⑤ — 자동 테스트는 반복 검증과 차이 발견에 효과적이다.

7 디버깅 로그

시간·입력·상태·오류를 기록

개념 상세 설명: 디버깅 로그는 프로그램 실행 중 중요한 상태와 오류 정보를 기록하는 자료이다. 단순한 출력보다 시간, 함수 이름, 입력값, 처리 단계와 오류 내용을 일정한 형식으로 남기면 원인을 찾기 쉽다.

import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)s | %(message)s"
)

def divide(a, b):
    logging.info("divide 시작: a=%s, b=%s", a, b)

    if b == 0:
        logging.error("0으로 나눌 수 없음")
        return None

    result = a / b
    logging.info("divide 완료: result=%s", result)
    return result

divide(10, 2)
divide(10, 0)
코드 상세 설명: 정상 실행은 INFO, 문제 상황은 ERROR 수준으로 기록한다. 입력과 결과가 로그에 남아 오류가 발생한 조건을 확인할 수 있다.
[개념 7] 디버깅 로그에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ⑤ — 로그는 일정한 형식으로 상태와 오류를 남기되 민감정보는 보호해야 한다.

8 오류 재현성과 최소 재현 사례

같은 조건에서 같은 오류 만들기

개념 상세 설명: 오류 재현성은 같은 입력, 실행 순서와 환경에서 같은 오류를 다시 만들 수 있는 성질이다. 오류를 일으키는 가장 작은 입력과 코드만 남긴 최소 재현 사례를 만들면 원인을 더 쉽게 찾을 수 있다.

def first_item(values):
    return values[0]

# 최소 재현 사례
sample = []

try:
    print(first_item(sample))
except IndexError as error:
    print("재현된 오류:", type(error).__name__)
    print("재현 입력:", sample)
    print("원인: 빈 목록의 첫 항목 접근")
코드 상세 설명: 복잡한 전체 프로그램 대신 빈 목록 하나로 IndexError가 발생하는 핵심 조건만 남겼다. 다른 사람도 같은 입력으로 오류를 확인할 수 있다.
[개념 8] 오류 재현성에 대한 옳은 설명 3개를 고르시오.
정답: ①, ②, ④ — 재현 가능한 입력·환경·절차를 기록하고 최소 사례로 단순화한다.

9 원인 분석과 수정 확인

증상과 원인을 구분한다

개념 상세 설명: 오류 메시지는 증상일 수 있으며 실제 원인은 그보다 앞선 단계에 있을 수 있다. 가설을 세우고 중간값을 확인한 뒤 한 번에 한 가지를 수정하고 같은 테스트로 결과를 다시 확인한다.

def mean(values):
    if len(values) == 0:
        return None
    return sum(values) / len(values)

# 수정 전에는 빈 목록에서 ZeroDivisionError 가능
tests = [
    ([10, 20], 15.0),
    ([], None)
]

for data, expected in tests:
    actual = mean(data)
    print({
        "입력": data,
        "예상": expected,
        "실제": actual,
        "통과": actual == expected
    })
코드 상세 설명: 빈 목록을 먼저 처리하도록 수정하고, 오류를 재현했던 입력을 다시 사용해 수정 여부를 확인한다. 정상 입력도 함께 시험해 기존 기능이 유지되는지 살핀다.
[개념 9] 오류 수정 과정에 대한 옳은 설명 3개를 고르시오.
정답: ②, ③, ⑤ — 원인을 단계적으로 확인하고 같은 사례로 수정 효과를 검증한다.

10 회귀 테스트와 검증 기록

수정 때문에 다른 기능이 깨지지 않았는지 확인

개념 상세 설명: 회귀 테스트는 오류를 수정하거나 기능을 추가한 뒤 기존 기능이 여전히 정상 동작하는지 확인하는 테스트이다. 테스트 결과, 코드 버전, 실행 환경과 수정 내역을 기록하면 검증의 재현성과 신뢰성이 높아진다.

APP_VERSION = "1.2.0"

def normalize(value):
    if value is None:
        return None
    return max(0, min(float(value), 100))

regression_tests = [
    (50, 50.0),
    (-5, 0),
    (120, 100),
    (None, None)
]

passed = 0

for value, expected in regression_tests:
    actual = normalize(value)
    ok = actual == expected
    passed += ok
    print(APP_VERSION, value, expected, actual, ok)

print(f"통과: {passed}/{len(regression_tests)}")
코드 상세 설명: 정상, 경계 밖 값과 결측값을 다시 시험한다. 코드 버전을 함께 기록하여 어느 상태에서 테스트했는지 확인할 수 있다.
[개념 10] 회귀 테스트와 기록에 대한 옳은 설명 3개를 고르시오.
정답: ①, ④, ⑤ — 수정의 부작용을 확인하고 버전·환경·결과를 남긴다.

1. 경계값 테스트가 입력 검증 오류 발견에 미치는 효과

구현: 점수, 온도 또는 센서 범위 검사 함수를 제작하고 중앙값 중심 테스트와 경계값 중심 테스트를 비교한다.

탐구 관점: 발견한 오류 수, 테스트 수와 수정 시간을 분석한다.

구현: 로그 없음, 단순 출력, 구조화된 로그를 사용하는 세 버전을 만든다.

탐구 관점: 오류 위치를 찾는 시간과 재현 성공률을 비교한다.

구현: 함수별 정상·경계·예외 자동 테스트를 구성한다.

탐구 관점: 코드 수정 횟수에 따른 재발 오류와 회귀 오류의 변화를 분석한다.

4. 최소 재현 사례가 협업 디버깅에 미치는 영향

구현: 전체 프로젝트 오류 설명과 최소 재현 코드 설명을 각각 제공한다.

탐구 관점: 동료가 오류를 재현하고 원인을 찾는 데 걸린 시간을 비교한다.

5. 테스트 기록 형식에 따른 재현성 비교

구현: 자유 서술 기록과 입력·환경·절차·결과를 갖춘 표준 기록 양식을 만든다.

탐구 관점: 다른 사용자의 재현 성공률과 누락 정보의 차이를 분석한다.
