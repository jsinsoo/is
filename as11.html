1 프로토타입과 최소 기능

작게 만들고 빠르게 확인

개념 상세 설명: 프로토타입은 아이디어와 핵심 기능의 가능성을 확인하기 위해 만든 초기 시제품이다. 모든 기능과 디자인을 구현하기보다 가장 중요한 입력, 처리와 출력이 연결되는지 먼저 검증한다.

prototype_features = {
    "자료 입력": True,
    "결측값 제거": True,
    "평균 계산": True,
    "그래프 출력": False,
    "사용자 로그인": False
}

for feature, included in prototype_features.items():
    if included:
        print("1차 프로토타입 포함:", feature)
코드 상세 설명: 첫 프로토타입에 꼭 필요한 기능만 True로 지정한다. 부가 기능은 뒤 단계로 미루어 핵심 흐름부터 검증한다.
[개념 1] 프로토타입에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ⑤ — 프로토타입은 핵심 기능을 빠르게 구현하고 검증하는 데 목적이 있다.

2 함수의 역할과 반환값

입력을 받아 결과를 돌려준다

개념 상세 설명: 함수는 특정 기능을 이름 붙여 묶은 코드 단위이다. 매개변수로 자료를 받고 처리한 결과를 return으로 반환하면 다른 기능과 쉽게 연결하고 시험할 수 있다.

def calculate_mean(values):
    if not values:
        return None
    return sum(values) / len(values)

result = calculate_mean([12, 15, 18])
print("평균:", result)
코드 상세 설명: 리스트가 비어 있으면 None을 반환하고, 값이 있으면 평균을 반환한다. 출력만 하는 함수보다 반환값이 있는 함수가 다른 처리와 연결하기 쉽다.
[개념 2] 함수에 대한 옳은 설명 3개를 고르시오.
정답: ①, ②, ④ — 함수는 입력을 받아 처리하고 결과를 반환하도록 설계할 수 있다.

3 한 함수 한 책임

역할을 작고 분명하게

개념 상세 설명: 한 함수는 가능한 한 한 가지 역할만 담당해야 한다. 입력, 정제, 계산과 출력이 모두 섞인 긴 함수보다 작은 함수 여러 개가 이해, 수정과 시험에 유리하다.

def remove_missing(values):
    return [value for value in values if value is not None]

def calculate_maximum(values):
    return max(values) if values else None

raw = [18, None, 24, 21]
clean = remove_missing(raw)
maximum = calculate_maximum(clean)

print("정제 자료:", clean)
print("최댓값:", maximum)
코드 상세 설명: 결측값 제거와 최댓값 계산을 서로 다른 함수로 분리한다. 한 기능의 변경이 다른 기능에 미치는 영향을 줄일 수 있다.
[개념 3] 한 함수 한 책임 원칙에 대한 옳은 설명 3개를 고르시오.
정답: ②, ③, ⑤ — 함수는 작고 분명한 책임을 갖도록 설계한다.

4 모듈화

관련 기능을 파일과 영역으로 구성

개념 상세 설명: 모듈화는 관련된 함수, 상수와 자료형을 의미 있는 단위로 묶는 설계 방식이다. 수집 모듈, 전처리 모듈과 분석 모듈처럼 나누면 역할과 연결 관계가 분명해진다.

# collector.py에 있다고 가정
def collect_sample():
    return [21.5, 22.0, 22.8, 23.1]

# analyzer.py에 있다고 가정
def summarize(values):
    return {
        "count": len(values),
        "mean": sum(values) / len(values)
    }

data = collect_sample()
summary = summarize(data)
print(summary)
코드 상세 설명: 예시는 한 화면에 작성했지만 실제 프로젝트에서는 수집과 분석 기능을 별도 파일로 나눌 수 있다. 각 모듈의 입력과 출력 형식을 일관되게 설계해야 한다.
[개념 4] 모듈화에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ④ — 모듈화는 관련 기능을 묶고 연결 규칙을 명확하게 한다.

5 작은 데이터 수집

전체 자료 전에 표본으로 확인

개념 상세 설명: 프로토타입 단계에서는 수천 개의 실제 자료보다 구조를 대표하는 작은 자료가 유용하다. 정상값, 결측값, 중복값과 범위 밖 값을 일부 포함해 수집 기능과 다음 처리 단계를 함께 점검한다.

sample_rows = [
    {"time": "09:00", "co2": 620},
    {"time": "10:00", "co2": 780},
    {"time": "11:00", "co2": None},
    {"time": "11:00", "co2": None},
    {"time": "12:00", "co2": 1450}
]

print("수집 행 수:", len(sample_rows))
for row in sample_rows:
    print(row)
코드 상세 설명: 정상 자료뿐 아니라 결측값과 중복 행을 의도적으로 넣었다. 작은 자료라도 실제 문제에서 발생할 수 있는 형태를 대표해야 한다.
[개념 5] 작은 데이터 활용에 대한 옳은 설명 3개를 고르시오.
정답: ①, ②, ⑤ — 작은 대표 자료는 초기 기능 검증에 효과적이지만 전체 성능을 보장하지는 않는다.

6 입력 형식 검증

처리 전에 자료형 확인

개념 상세 설명: 입력 검증은 자료가 예상한 형식과 자료형을 갖는지 확인하는 과정이다. 숫자로 변환할 수 없는 문자열, 빈 입력과 잘못된 날짜 형식을 처리하지 않으면 분석 단계에서 오류가 발생한다.

def parse_temperature(raw):
    if raw is None or str(raw).strip() == "":
        return None

    try:
        return float(raw)
    except (TypeError, ValueError):
        return None

samples = ["23.5", "", "오류", 18]

for sample in samples:
    print(sample, "→", parse_temperature(sample))
코드 상세 설명: 빈 값과 숫자로 변환할 수 없는 값을 None으로 통일한다. 다음 전처리 함수가 동일한 기준으로 결측값을 처리할 수 있다.
[개념 6] 입력 형식 검증에 대한 옳은 설명 3개를 고르시오.
정답: ②, ④, ⑤ — 입력 형식을 먼저 확인해야 이후 처리의 안정성이 높아진다.

7 범위 검증과 전처리

가능한 값만 남긴다

개념 상세 설명: 자료형이 숫자라고 해도 현실적으로 가능한 범위인지 확인해야 한다. 센서 특성, 측정 단위와 연구 조건에 맞는 허용 범위를 정하고 범위 밖 값은 제거, 보정 또는 별도 표시한다.

def clean_humidity(values):
    clean = []

    for value in values:
        if isinstance(value, (int, float)) and 0 <= value <= 100:
            clean.append(float(value))

    return clean

raw = [45, 52.5, -3, 108, None, 61]
print(clean_humidity(raw))
코드 상세 설명: 숫자이면서 0~100 범위 안에 있는 습도만 남긴다. 제거 기준은 임의로 정하지 말고 자료의 의미와 측정 조건을 근거로 정해야 한다.
[개념 7] 범위 검증과 전처리에 대한 옳은 설명 3개를 고르시오.
정답: ①, ③, ⑤ — 범위 검증은 자료의 의미와 측정 조건에 근거해야 한다.

8 분석 함수와 요약 결과

원자료를 의미 있는 정보로

개념 상세 설명: 분석 함수는 정제된 자료를 받아 평균, 최솟값, 최댓값과 자료 수처럼 탐구 질문에 필요한 결과를 계산한다. 반환 형식을 딕셔너리로 통일하면 출력과 저장 기능이 사용하기 쉽다.

def summarize(values):
    if not values:
        return {
            "count": 0,
            "mean": None,
            "minimum": None,
            "maximum": None
        }

    return {
        "count": len(values),
        "mean": sum(values) / len(values),
        "minimum": min(values),
        "maximum": max(values)
    }

print(summarize([18.2, 19.5, 21.0, 20.3]))
코드 상세 설명: 빈 자료도 오류 없이 처리하고 항상 같은 키를 가진 딕셔너리를 반환한다. 다른 모듈이 결과 구조를 예측할 수 있다.
[개념 8] 분석 함수 설계에 대한 옳은 설명 3개를 고르시오.
정답: ①, ②, ④ — 분석 함수는 정제 자료와 일관된 반환 형식을 사용한다.

9 단위 시험과 assert

작은 기능을 독립적으로 확인

개념 상세 설명: 단위 시험은 함수 하나가 예상대로 작동하는지 작은 입력과 예상 결과로 확인하는 방법이다. assert는 조건이 거짓일 때 오류를 발생시켜 잘못된 결과를 빠르게 발견한다.

def remove_missing(values):
    return [value for value in values if value is not None]

assert remove_missing([1, None, 3]) == [1, 3]
assert remove_missing([]) == []
assert remove_missing([None]) == []

print("모든 단위 시험 통과")
코드 상세 설명: 정상 입력, 빈 입력과 결측값만 있는 입력을 각각 시험한다. 함수가 수정된 뒤에도 같은 시험을 반복하면 기존 기능이 깨졌는지 확인할 수 있다.
[개념 9] 단위 시험에 대한 옳은 설명 3개를 고르시오.
정답: ②, ③, ⑤ — 단위 시험은 작은 기능을 예상 결과와 비교해 반복 검증한다.

10 프로토타입 파이프라인 연결

수집→검증→전처리→분석→출력

개념 상세 설명: 각 함수가 정상적으로 작동하면 전체 실행 흐름을 연결한다. 단계별 중간 결과를 확인하고 오류가 발생한 위치를 기록하면 프로토타입의 개선 지점을 찾기 쉽다.

def collect():
    return ["21.5", "", "오류", "23.0", "25.5"]

def parse(values):
    result = []
    for value in values:
        try:
            result.append(float(value))
        except ValueError:
            continue
    return result

def analyze(values):
    return sum(values) / len(values) if values else None

def run():
    raw = collect()
    clean = parse(raw)
    mean = analyze(clean)

    print("원자료:", raw)
    print("정제 자료:", clean)
    print("평균:", mean)

run()
코드 상세 설명: 수집, 변환과 분석 함수를 run()에서 순서대로 연결한다. 단계별 결과를 출력해 어느 단계에서 자료가 달라졌는지 확인한다.
[개념 10] 프로토타입 파이프라인에 대한 옳은 설명 3개를 고르시오.
정답: ①, ④, ⑤ — 함수 사이의 형식을 맞추고 전체 흐름을 단계별로 확인한다.

📚 학습과 관련한 탐구 보고서 주제 예시 5개

1. 교실 이산화탄소 자료 처리 프로토타입

구현: 소규모 센서 자료를 입력받아 결측값과 비정상값을 제거하고 평균과 최고 농도를 계산한다.

탐구 관점: 함수 분리 전후의 코드 이해도와 오류 수정 과정을 비교한다.

2. 학교 전력 사용량 요약 시스템의 최소 기능 구현

구현: 날짜와 사용량을 검증하고 일평균, 최댓값과 기준 초과일을 출력한다.

탐구 관점: 입력 검증 규칙이 결과 신뢰성에 미치는 영향을 분석한다.

3. 실험 측정값 이상 탐지 프로토타입

구현: 허용 범위와 평균에서의 거리 기준으로 의심 값을 표시한다.

탐구 관점: 작은 시험 자료의 구성 방식과 오류 발견률을 비교한다.

4. 급식 만족도 자료 전처리 모듈 개발

구현: 빈 응답, 범위 밖 점수와 중복 응답을 처리하는 함수를 만든다.

탐구 관점: 전처리 기준에 따른 평균과 분포 변화의 차이를 분석한다.

5. 기상 자료 수집·분석 함수의 단위 시험 연구

구현: 수집, 숫자 변환, 범위 검증과 요약 함수를 구현한다.

탐구 관점: 정상·경계·예외 시험이 프로토타입 오류 감소에 미치는 효과를 기록한다.
