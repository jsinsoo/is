1. 협업 통합의 핵심 절차

① 역할 분리입력·분석·출력·테스트처럼 책임이 겹치지 않게 모듈을 나눕니다.
② 인터페이스 합의함수 이름, 매개변수, 반환형, 오류 처리 규칙을 먼저 정합니다.
③ 독립 검증각 모듈을 단위 테스트로 확인하고 코드 리뷰를 진행합니다.
④ 버전 통합브랜치·커밋·병합 규칙을 사용해 변경 이력을 남기고 충돌을 해결합니다.
모듈 협업에서 가장 중요한 것은 내부 구현보다 서로 연결되는 경계입니다. 함수가 어떤 입력을 받고 무엇을 반환하는지 합의하면 각 팀원이 독립적으로 개발해도 통합이 쉬워집니다.

오류 처리잘못된 입력에서 예외를 발생시킬지 기본값을 반환할지 합의했는가?

버전 관리기능별 브랜치와 의미 있는 커밋 메시지를 사용하는가?

예제 2

함수 인터페이스 명세 작성

타입 힌트docstring
def classify_co2(value: float) -> str:
    """
    이산화탄소 농도를 상태 문자열로 변환한다.

    매개변수:
        value: 0 이상의 ppm 값

    반환값:
        "정상", "환기 권장", "즉시 환기"

    예외:
        value가 음수이면 ValueError
    """
    if value < 0:
        raise ValueError(
            "농도는 0 이상이어야 합니다."
        )

    if value >= 1500:
        return "즉시 환기"

    if value >= 1000:
        return "환기 권장"

    return "정상"
인터페이스 합의
입력 자료형, 반환 자료형, 허용 범위, 예외 규칙을 먼저 정하면 다른 모듈이 같은 기준으로 연결됩니다.
예제 3

데이터 입력 모듈 작성

data_io.py검증
# data_io.py

def parse_values(text: str) -> list[float]:
    values = []

    for item in text.split(","):
        item = item.strip()

        if not item:
            continue

        value = float(item)

        if value < 0:
            raise ValueError(
                "음수 데이터는 허용하지 않습니다."
            )

        values.append(value)

    if not values:
        raise ValueError(
            "입력 데이터가 없습니다."
        )

    return values
책임
입력 모듈은 문자열을 숫자 목록으로 변환하고 잘못된 값만 검증합니다. 평균 계산이나 상태 판정은 다른 모듈에 맡깁니다.
예제 4

분석 모듈 작성

analyzer.py순수 함수
# analyzer.py

from statistics import mean

def summarize(
    values: list[float]
) -> dict[str, float]:
    if not values:
        raise ValueError(
            "분석할 값이 없습니다."
        )

    return {
        "평균": mean(values),
        "최댓값": max(values),
        "최솟값": min(values)
    }

def count_warning(
    values: list[float],
    threshold: float = 1000
) -> int:
    return sum(
        1
        for value in values
        if value >= threshold
    )
순수 함수
외부 파일이나 전역 변수에 의존하지 않고 입력만으로 결과를 만들면 테스트와 재사용이 쉬워집니다.
예제 5

출력 모듈 인터페이스 작성

reporter.py문자열 반환
# reporter.py

def make_report(
    summary: dict[str, float],
    warning_count: int
) -> str:
    lines = [
        "[분석 결과]",
        f"평균: {summary['평균']:.1f}",
        f"최댓값: {summary['최댓값']:.1f}",
        f"최솟값: {summary['최솟값']:.1f}",
        f"경고 횟수: {warning_count}"
    ]

    return "\n".join(lines)
통합 친화성
출력 모듈이 직접 print()하지 않고 문자열을 반환하면 콘솔·파일·웹 화면 등 여러 환경에서 재사용할 수 있습니다.
예제 6

모듈 간 예외 처리 규칙 통일

예외오류 계약
def safe_execute(text: str) -> str:
    try:
        values = parse_values(text)
        summary = summarize(values)
        warnings = count_warning(values)

        return make_report(
            summary,
            warnings
        )

    except ValueError as error:
        return (
            "[입력 오류]\n"
            + str(error)
        )

print(safe_execute("720, 860, 1040"))
print()
print(safe_execute("720, -1, 900"))
규칙
모든 모듈이 제각각 오류 문장을 출력하지 않고, 핵심 모듈은 예외를 발생시키고 통합 모듈에서 사용자용 문장으로 변환합니다.
예제 7

단위 테스트로 인터페이스 검증

unittest경계값
import unittest

class TestAnalyzer(unittest.TestCase):
    def test_summary(self):
        result = summarize(
            [10, 20, 30]
        )

        self.assertEqual(
            result["평균"],
            20
        )

        self.assertEqual(
            result["최댓값"],
            30
        )

    def test_warning_count(self):
        self.assertEqual(
            count_warning(
                [900, 1000, 1200]
            ),
            2
        )

    def test_empty_values(self):
        with self.assertRaises(ValueError):
            summarize([])

if __name__ == "__main__":
    unittest.main()
검증 기준
정상 입력뿐 아니라 임계값 1000, 빈 목록, 음수 입력처럼 경계값과 오류 상황도 검사합니다.
예제 8

리뷰체크리스트
review = {
    "함수 이름이 역할을 설명하는가": True,
    "타입 힌트가 있는가": True,
    "중복 코드가 제거되었는가": False,
    "예외 처리가 일관적인가": True,
    "테스트가 포함되었는가": True,
    "출처와 라이선스가 기록되었는가": True
}

failed = []

for item, passed in review.items():
    status = (
        "통과"
        if passed
        else "보완"
    )

    print(item, ":", status)

    if not passed:
        failed.append(item)

print("보완 항목 수:", len(failed))
리뷰 관점

예제 10

종합: 모듈 통합 실행 프로그램

main.py통합버전 정보
# main.py

from dataclasses import dataclass

@dataclass
class AppConfig:
    warning_threshold: float = 1000
    version: str = "1.0.0"

def run(
    text: str,
    config: AppConfig
) -> str:
    values = parse_values(text)

    summary = summarize(values)

    warning_count = count_warning(
        values,
        config.warning_threshold
    )

    report = make_report(
        summary,
        warning_count
    )

    return (
        f"버전: {config.version}\n"
        + report
    )

if __name__ == "__main__":
    config = AppConfig(
        warning_threshold=1000,
        version="1.0.0"
    )

    sample = "720, 860, 1040, 1280"
    print(run(sample, config))
통합 단계담당확인 내용완료 기준
인터페이스 명세전체 팀함수명·입력·반환·예외 규칙공통 문서 승인
모듈 개발역할별 담당자한 모듈 한 책임단위 테스트 통과
코드 리뷰다른 팀원정확성·가독성·중복·보안보완 의견 반영
브랜치 병합통합 담당충돌 해결·최신 main 반영병합 커밋 생성
통합 테스트검증 담당모듈 연결·경계값·오류 입력전체 테스트 통과
릴리스문서 담당버전·변경 이력·사용법태그와 배포 문서 작성
협업 과정 설명 예시
“팀원별로 데이터 입력, 분석, 결과 출력 모듈을 나누어 개발하였다. 통합 전에 함수 이름, 매개변수, 반환 자료형, 예외 처리 규칙을 공통 인터페이스로 정의하였다. 각 모듈은 단위 테스트와 상호 코드 검토를 거쳤으며, 기능별 브랜치에서 작업한 뒤 메인 브랜치의 최신 변경을 반영하여 병합하였다. 통합 후에는 정상 입력과 경계값, 오류 입력을 대상으로 전체 테스트를 수행하고 버전과 변경 이력을 기록하였다.”

마무리 진단 문제

1. 모듈 인터페이스에 포함해야 할 핵심 정보는 무엇인가?
정답 보기함수 이름, 매개변수, 자료형, 반환값, 허용 범위, 예외 처리 규칙입니다.
2. 한 모듈이 한 가지 책임에 집중해야 하는 이유는 무엇인가?
정답 보기수정 영향 범위를 줄이고 테스트·재사용·역할 분담을 쉽게 하기 위해서입니다.
3. 단위 테스트와 통합 테스트의 차이는 무엇인가?
정답 보기단위 테스트는 개별 함수나 모듈을 확인하고, 통합 테스트는 여러 모듈이 연결된 전체 흐름을 확인합니다.
4. 기능별 브랜치를 사용하는 장점은 무엇인가?
정답 보기메인 코드에 영향을 주지 않고 독립적으로 개발·검토한 뒤 안전하게 병합할 수 있습니다.
5. 코드 리뷰에서 기능 동작 외에 확인해야 할 항목은 무엇인가?
정답 보기가독성, 중복 코드, 예외 처리, 테스트, 보안, 라이선스와 출처 표시 등을 확인해야 합니다.
