예제 1

교실 환기 알림 프로젝트 발표

환경센서
항목발표 내용
문제 정의수업 후반 교실 공기 질 저하를 줄이기 위해 환기 시점을 자동 안내함.
설계이산화탄소 농도를 입력받아 정상·주의·즉시 환기로 출력함.
알고리즘임계값 조건문과 최근 3개 측정값의 이동평균을 사용함.
검증999, 1000, 1499, 1500ppm 경계값을 테스트함.
윤리학생 이름과 위치 정보는 수집하지 않고 교실 단위 집계값만 사용함.
def ventilation_status(co2):
    if co2 >= 1500:
        return "즉시 환기"
    if co2 >= 1000:
        return "환기 권장"
    return "정상"

tests = [999, 1000, 1499, 1500]

for value in tests:
    print(value, ventilation_status(value))
개인 성찰
처음에는 센서값 한 번만으로 상태를 판정했지만 순간 오차가 크다는 점을 발견하였다. 이동평균을 적용하면서 실제 데이터의 흔들림을 줄이는 방법을 배웠다.
예제 2

급식 대기시간 분석 발표

큐학교생활

발표 핵심

문제: 특정 시간대의 긴 대기열

설계: 학생 도착 순서를 큐로 저장

검증: 실제 관찰값과 예상 대기시간 비교

윤리: 익명 번호만 사용

def waiting_times(service_times):
    current = 0
    result = []

    for service in service_times:
        result.append(current)
        current += service

    return result

times = waiting_times([2, 3, 2, 4])

print("개별 대기:", times)
print("평균:", sum(times) / len(times))
성찰
큐 자료구조가 실제 급식 대기 상황을 자연스럽게 표현한다는 점을 이해하였다. 이후에는 배식대 수가 여러 개인 경우를 모델링하고 싶다.
예제 3

도서 추천 프로그램 발표

추천편향
항목발표 내용
문제 정의학생이 관심 분야의 책을 빠르게 찾기 어려운 문제
알고리즘선호 장르 필터링 후 평점 내림차순 정렬
검증장르별 추천 결과와 사용자 만족도 설문 비교
윤리대출 기록 익명화, 인기 도서만 반복 추천되는 편향 점검
books = [
    ("파이썬 기초", "정보", 4.8),
    ("알고리즘 여행", "정보", 4.6),
    ("우주의 구조", "과학", 4.9)
]

def recommend(genre):
    candidates = [
        book
        for book in books
        if book[1] == genre
    ]

    return sorted(
        candidates,
        key=lambda book: book[2],
        reverse=True
    )

print(recommend("정보"))
성찰
평점만으로 추천하면 소수 분야의 책이 배제될 수 있음을 알게 되었다. 추천의 정확성뿐 아니라 다양성과 공정성도 중요한 기준이라는 점을 배웠다.
예제 4

교내 최단 경로 안내 발표

그래프BFS
from collections import deque

def shortest_path(graph, start, target):
    queue = deque([(start, [start])])
    visited = {start}

    while queue:
        node, path = queue.popleft()

        if node == target:
            return path

        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append(
                    (neighbor, path + [neighbor])
                )

    return None
발표 항목설명
설계장소를 노드, 통로를 간선으로 표현함.
알고리즘모든 이동 비용이 같아 BFS를 선택함.
검증출발지와 도착지가 같을 때, 경로가 없을 때를 테스트함.
성찰문제를 그래프로 바꾸면 실제 공간 문제를 알고리즘으로 해결할 수 있음을 이해함.

power = [42, 55, 61, 64, 58, 51, 63, 60, 44]

average = sum(power) / len(power)

high_usage = [
    value
    for value in power

print("평균 초과 횟수:", len(high_usage))

윤리

예제 7

미세먼지 실외 활동 추천 발표

공공데이터윤리적 판단
def activity_status(pm25, rain):
    if rain:
        return "실내 활동 권장"

    if pm25 > 35:
        return "미세먼지 주의"

    return "실외 활동 가능"

print(activity_status(28, False))
print(activity_status(42, False))
윤리적 고려
건강 관련 최종 결정은 프로그램이 아니라 교사와 공식 기준이 해야 하며, 최신 기준을 확인하지 않은 값을 단정적으로 사용해서는 안 됩니다.
성찰
프로그램의 결과가 사람의 행동에 영향을 줄 수 있으므로 정확성 외에 책임 있는 표현이 필요하다는 점을 배웠다.
예제 8

교실 좌석 배치 프로젝트 발표

백트래킹개인정보
students = ["A", "B", "C"]
selected = []

def arrange():
    if len(selected) == len(students):
        print(selected)
        return

    for student in students:
        if student in selected:
            continue

        selected.append(student)
        arrange()
        selected.pop()

arrange()
항목발표 내용
설계학생을 익명 기호로 바꾸고 가능한 배치를 생성함.
알고리즘선택·재귀·되돌리기 방식의 백트래킹 사용
검증중복 배치 여부와 조건 위반 배치가 제거되는지 확인
윤리시력·건강 정보 등 민감 정보는 공개 발표에 포함하지 않음
예제 9

공용 프린터 큐 관리 발표

큐효율성
from collections import deque

jobs = deque([
    ("보고서", 10),
    ("발표자료", 6),
    ("과제", 4)
])

time = 0

while jobs:
    name, pages = jobs.popleft()
    print(name, "시작:", time)
    time += pages
    print(name, "완료:", time)
성찰
선입선출 방식은 공정하지만 긴 작업 하나가 전체 대기를 늘릴 수 있음을 알게 되었다. 이후에는 페이지 수가 짧은 작업과 일반 작업을 분리하는 정책을 비교하고 싶다.

평가 항목발표에서 확인할 내용
문제 정의사용자와 문제 상황이 구체적인가?
설계입력·출력·기능·제한 조건이 연결되는가?
알고리즘자료구조와 알고리즘 선택 이유가 타당한가?
구현핵심 코드가 모듈화되고 이해하기 쉬운가?
검증정상·경계·예외 테스트와 결과 근거가 있는가?
윤리개인정보·편향·출처·라이선스를 검토했는가?
성찰배운 점과 개선 방향이 구체적인가?
개인 학습 성찰 예시
“이번 프로젝트를 통해 문제를 바로 코딩하기보다 사용자와 입력·출력·제한 조건을 먼저 정의하는 과정이 중요하다는 점을 배웠다. 초기 구현에서는 정상 입력만 확인했으나 경계값과 예외 입력에서 오류가 발생했고, 이를 재현 가능한 테스트로 기록하면서 검증의 필요성을 이해하였다. 또한 데이터 처리 과정에서 개인정보 최소 수집과 알고리즘 편향, 외부 자료의 출처와 라이선스를 함께 고려해야 함을 알게 되었다. 다음 프로젝트에서는 기능 구현과 동시에 테스트와 윤리 점검표를 작성하고, 사용자 피드백을 반영하여 더 신뢰할 수 있는 산출물을 만들고자 한다.”

마무리 진단 문제

1. 프로젝트 발표에서 문제 정의에 포함해야 할 내용은 무엇인가?
정답 보기사용자, 문제 상황, 해결 목표, 필요성을 포함해야 합니다.
2. 알고리즘 선택 이유를 설명할 때 필요한 것은 무엇인가?
정답 보기자료의 특성, 시간 복잡도, 적용 조건, 다른 방법과의 비교 근거입니다.
3. 검증 발표에 포함해야 할 테스트 유형은 무엇인가?
정답 보기정상 입력, 경계 입력, 예외 입력, 이전 오류에 대한 회귀 테스트입니다.
4. 윤리적 고려에 포함되는 대표 요소는 무엇인가?
정답 보기개인정보 보호, 알고리즘 편향, 저작권, 라이선스, 결과 사용의 책임입니다.
5. 좋은 학습 성찰은 어떤 내용을 포함하는가?
정답 보기잘한 점, 어려움, 해결 과정, 새롭게 배운 점, 다음 개선 목표를 구체적으로 포함합니다.
